## Пользовательские сценарии и тестирование *(обязательно)*

### Пользовательская история 1 — Игровая сессия с AI Dungeon Master (Приоритет: P1)

Бэкенд получает действия игроков через WebSocket, оркестрирует AI для генерации ответов мастера, извлекает изменения состояния мира и транслирует результаты всем игрокам в комнате в реальном времени.

**Почему этот приоритет**: Это основная функциональность бэкенда — AI-оркестрация и управление состоянием игры в реальном времени. Без этого игра невозможна.

**Независимый тест**: Можно протестировать, отправив действие игрока через WebSocket и проверив, что AI-ответ сгенерирован, состояние извлечено и ответ транслирован всем подключённым клиентам.

**Сценарии приёмки**:

1. **Дано** активная игровая сессия с подключёнными игроками, **Когда** получено действие игрока, **Тогда** бэкенд генерирует ответ AI-мастера в течение 10 секунд
2. **Дано** сгенерирован ответ AI-мастера, **Когда** завершается извлечение состояния, **Тогда** состояние мира (флаги, локация, события) обновляется в хранилище
3. **Дано** ответ мастера готов, **Когда** запускается трансляция, **Тогда** все игроки в комнате получают ответ одновременно через WebSocket
4. **Дано** ответ мастера требует бросок кубиков, **Когда** ответ парсится, **Тогда** структурированный запрос броска (тип, модификатор, СЛ, навык) включается в трансляцию
5. **Дано** получено действие игрока, **Когда** начинается обработка, **Тогда** действие сохраняется в историю сообщений до вызова AI

---

### Пользовательская история 2 — Генерация и управление сценариями (Приоритет: P2)

Бэкенд получает описания сценариев на естественном языке, оркестрирует AI для генерации структурированного JSON сценария, валидирует логику сценария и управляет историей версий.

**Почему этот приоритет**: Сценарии требуются для игры. Бэкенд должен обрабатывать логику AI-генерации и валидации.

**Независимый тест**: Можно протестировать, отправив промпт сценария и проверив возврат структурированного JSON с актами, NPC и локациями.

**Сценарии приёмки**:

1. **Дано** запрос на создание сценария с описанием, **Когда** AI-генерация завершается, **Тогда** возвращается структурированный сценарий с актами, NPC, локациями и world_state
2. **Дано** существующий сценарий, **Когда** получен промпт уточнения, **Тогда** перегенерируются только затронутые части сценария
3. **Дано** сгенерированный сценарий, **Когда** запускается валидация, **Тогда** обнаруживаются и сообщаются логические проблемы (недостижимые акты, неопределённые ссылки на NPC)
4. **Дано** редактирование сценария, **Когда** сохраняется, **Тогда** создаётся новая версия, а предыдущие версии остаются доступными
5. **Дано** запрос на восстановление версии, **Когда** обрабатывается, **Тогда** указанная версия становится текущей

---

### Пользовательская история 3 — Аутентификация и управление пользователями (Приоритет: P3)

Бэкенд обрабатывает регистрацию пользователей, вход, верификацию Sign in with Apple, управление JWT-токенами и операции с профилем.

**Почему этот приоритет**: Аутентификация — необходимая инфраструктура для всех других функций, но следует стандартным паттернам.

**Независимый тест**: Можно протестировать, зарегистрировав пользователя, получив токен и использовав его для доступа к защищённым эндпоинтам.

**Сценарии приёмки**:

1. **Дано** запрос на регистрацию с email/паролем, **Когда** валидация проходит, **Тогда** создаётся пользователь и возвращается JWT-токен
2. **Дано** токен Sign in with Apple, **Когда** верифицируется на серверах Apple, **Тогда** пользователь создаётся или сопоставляется и возвращается JWT
3. **Дано** валидный JWT-токен, **Когда** включён в заголовок запроса, **Тогда** запрос аутентифицируется и контекст пользователя доступен
4. **Дано** истёкший JWT-токен, **Когда** включён в запрос, **Тогда** возвращается 401 Unauthorized с понятным сообщением об ошибке
5. **Дано** аутентифицированный пользователь, **Когда** запрошено обновление профиля, **Тогда** данные пользователя обновляются и возвращается новое состояние

---

### Пользовательская история 4 — Управление игровыми комнатами и лобби (Приоритет: P4)

Бэкенд управляет жизненным циклом игровых комнат: создание, вход/выход игроков, статус готовности и переход к старту игры.

**Почему этот приоритет**: Координация мультиплеера важна, но может быть реализована после основной игровой логики.

**Независимый тест**: Можно протестировать, создав комнату, присоединив игроков и перейдя к старту игры.

**Сценарии приёмки**:

1. **Дано** запрос на создание комнаты с ID сценария, **Когда** обрабатывается, **Тогда** комната создаётся с хостом как первым игроком и статусом "waiting"
2. **Дано** открытая комната, **Когда** получен запрос на вступление, **Тогда** хост уведомляется и может принять/отклонить через API
3. **Дано** комната со всеми готовыми игроками, **Когда** хост запускает игру, **Тогда** создаётся игровая сессия и статус комнаты меняется на "active"
4. **Дано** активная комната, **Когда** запрашивается, **Тогда** возвращается текущий список игроков, статусы готовности и информация о сценарии
5. **Дано** отключение игрока, **Когда** обнаруживается, **Тогда** статус игрока обновляется, но комната продолжает работать для оставшихся игроков

---

### Пользовательская история 5 — Управление персонажами (Приоритет: P5)

Бэкенд хранит и извлекает персонажей игроков с атрибутами D&D 5e, валидирует данные персонажей и связывает персонажей с пользователями.

**Почему этот приоритет**: CRUD персонажей прост и может быть реализован с базовой валидацией.

**Независимый тест**: Можно протестировать, создав персонажа и использовав его в игровой комнате.

**Сценарии приёмки**:

1. **Дано** запрос на создание персонажа, **Когда** валидация D&D 5e проходит, **Тогда** персонаж сохраняется и возвращается с ID
2. **Дано** пользователь с персонажами, **Когда** запрашивается список персонажей, **Тогда** возвращаются все персонажи пользователя
3. **Дано** запрос на обновление персонажа, **Когда** обрабатывается, **Тогда** данные персонажа обновляются и валидация перезапускается
4. **Дано** персонаж назначен в активную игру, **Когда** запрошено удаление, **Тогда** удаление блокируется с соответствующей ошибкой

---

### Пользовательская история 6 — Прокси для голосовой обработки (Приоритет: P6)

Бэкенд проксирует TTS/STT запросы к внешним сервисам, управляет хранением аудиофайлов и обрабатывает потоковую доставку аудио.

**Почему этот приоритет**: Голос — премиум-функция; обработка на устройстве покрывает MVP. Бэкенд-прокси включает облачные голоса после MVP.

**Независимый тест**: Можно протестировать, отправив текст для TTS и получив URL аудио, или отправив аудио для STT и получив транскрипцию.

**Сценарии приёмки**:

1. **Дано** запрос TTS с текстом, **Когда** обрабатывается, **Тогда** генерируется аудио и возвращается URL
2. **Дано** запрос STT с аудиофайлом, **Когда** обрабатывается, **Тогда** возвращается транскрибированный текст
3. **Дано** сгенерированное аудио, **Когда** сохраняется, **Тогда** аудио доступно по CDN URL в течение 24 часов
4. **Дано** длинный текст для TTS, **Когда** поддерживается стриминг, **Тогда** аудио-чанки доставляются постепенно

### Граничные случаи

- Что происходит, когда AI-провайдер недоступен или ограничен по rate limit?
  - Бэкенд возвращает 503 с заголовком retry-after; клиент показывает "Мастер отдыхает"
- Как система обрабатывает разрывы WebSocket-соединения посреди игры?
  - Состояние сессии сохраняется; переподключившийся клиент получает полную историю сообщений с момента отключения
- Что происходит, когда генерация сценария превышает лимит токенов?
  - Бэкенд обрезает или разделяет запрос; возвращает частичный сценарий с предупреждением
- Как система обрабатывает одновременные действия нескольких игроков?
  - Действия ставятся в очередь с метками времени; обрабатываются последовательно; каждый ответ ссылается на инициирующее действие
- Что происходит при сбое извлечения состояния?
  - Игра продолжается с предыдущим состоянием; ошибка логируется; ручная коррекция состояния доступна через админ-эндпоинт
- Как система обрабатывает вредоносный или неуместный контент в промптах?
  - Применяются фильтры безопасности AI-провайдера; дополнительные пользовательские правила в системном промпте; помеченный контент возвращает 422 с причиной

## Требования *(обязательно)*

### Функциональные требования

**Сервис аутентификации**
- **FR-001**: API ДОЛЖЕН поддерживать регистрацию пользователя с email и паролем
  - Пароли ДОЛЖНЫ хэшироваться алгоритмом Argon2id (OWASP рекомендация)
  - Минимальная длина пароля: 8 символов
- **FR-002**: API ДОЛЖЕН верифицировать identity-токены Sign in with Apple
- **FR-003**: API ДОЛЖЕН выдавать JWT-токены с настраиваемым временем истечения
- **FR-004**: API ДОЛЖЕН валидировать JWT-токены на защищённых эндпоинтах
- **FR-005**: API ДОЛЖЕН поддерживать обновление токена без повторной аутентификации

**Сервис пользователей**
- **FR-006**: API ДОЛЖЕН хранить профили пользователей (email, имя, URL аватара)
- **FR-007**: API ДОЛЖЕН поддерживать получение и обновление профиля
- **FR-008**: API ДОЛЖЕН обеспечивать уникальность email

**Сервис персонажей**
- **FR-009**: API ДОЛЖЕН хранить персонажей с классом, расой, характеристиками и предысторией
- **FR-010**: API ДОЛЖЕН валидировать атрибуты персонажа по правилам D&D 5e:
  - Каждая характеристика: 1-20 (после расовых бонусов)
  - Сумма базовых характеристик: 60-90 (стандартный массив или point buy эквивалент)
  - Валидные классы: barbarian, bard, cleric, druid, fighter, monk, paladin, ranger, rogue, sorcerer, warlock, wizard
  - Валидные расы: dragonborn, dwarf, elf, gnome, half-elf, halfling, half-orc, human, tiefling
- **FR-011**: API ДОЛЖЕН поддерживать CRUD-операции с персонажами
- **FR-012**: API ДОЛЖЕН связывать персонажей с владеющим пользователем

**Сервис сценариев**
- **FR-013**: API ДОЛЖЕН генерировать структурированные сценарии из естественного языка через AI
- **FR-014**: API ДОЛЖЕН валидировать логику сценария (достижимые акты, определённые ссылки)
- **FR-015**: API ДОЛЖЕН хранить сценарии с историей версий
- **FR-016**: API ДОЛЖЕН поддерживать получение, обновление и восстановление версий сценария
- **FR-017**: API ДОЛЖЕН поддерживать итеративное уточнение сценария через дополнительные промпты

**Сервис лобби**
- **FR-018**: API ДОЛЖЕН поддерживать создание комнаты с назначением сценария
- **FR-019**: API ДОЛЖЕН поддерживать листинг комнат с фильтрами (открытые, по сценарию, по хосту)
- **FR-020**: API ДОЛЖЕН управлять запросами на вступление и одобрением хоста
- **FR-021**: API ДОЛЖЕН отслеживать статус готовности игроков
- **FR-022**: API ДОЛЖЕН обеспечивать лимит игроков в комнате (2-5 игроков)

  - При старте игры с <2 игроками: возвращать 400 Bad Request с сообщением "Недостаточно игроков для старта"

**Сервис сессий**
- **FR-023**: API ДОЛЖЕН создавать игровые сессии при старте комнаты
- **FR-024**: API ДОЛЖЕН поддерживать WebSocket-соединения для всех игроков в сессии
- **FR-025**: API ДОЛЖЕН сохранять все сообщения сессии с метками времени
- **FR-026**: API ДОЛЖЕН транслировать сообщения всем игрокам в комнате
- **FR-027**: API ДОЛЖЕН отслеживать состояние мира сессии (текущий акт, локация, флаги)

**Сервис AI-оркестрации**

- **FR-030**: API ДОЛЖЕН реализовать кэширование промптов для оптимизации затрат
- **FR-031**: API ДОЛЖЕН ограничивать историю сообщений последними 15 сообщениями в контексте
- **FR-032**: API ДОЛЖЕН парсить ответы AI на предмет запросов бросков кубиков

**Сервис извлечения состояния**
- **FR-033**: API ДОЛЖЕН извлекать изменения состояния мира из ответов мастера через AI
- **FR-034**: API ДОЛЖЕН использовать экономичную AI-модель для извлечения состояния
- **FR-035**: API ДОЛЖЕН обновлять состояние мира в кэше и базе данных
- **FR-036**: API ДОЛЖЕН определять завершение сцены и переходы между актами

**Сервис голоса**
- **FR-037**: API ДОЛЖЕН проксировать TTS-запросы к внешнему провайдеру
- **FR-038**: API ДОЛЖЕН проксировать STT-запросы к внешнему провайдеру
- **FR-039**: API ДОЛЖЕН хранить сгенерированное аудио с временными URL
- **FR-040**: API ДОЛЖЕН поддерживать стриминг аудио для ответов >500 символов (chunked transfer encoding)

**Модерация контента**
- **FR-041**: API ДОЛЖЕН применять рекомендации безопасности AI-провайдера через системный промпт
- **FR-042**: API ДОЛЖЕН фильтровать пользовательские промпты на запрещённый контент
- **FR-043**: API ДОЛЖЕН логировать события модерации для ревью

**Observability**
- **FR-044**: API ДОЛЖЕН выводить структурированные логи в формате JSON (через structlog)
- **FR-045**: API ДОЛЖЕН логировать метаданные запроса/ответа (эндпоинт, длительность, код статуса, user_id)
- **FR-046**: API ДОЛЖЕН экспонировать метрики latency по эндпоинтам (p50, p95, p99) через эндпоинт /metrics (Prometheus-формат)
- **FR-047**: API ДОЛЖЕН экспонировать метрики error rate по эндпоинтам через эндпоинт /metrics (Prometheus-формат)
- **FR-048**: API ДОЛЖЕН включать correlation ID (X-Request-ID) в логи для трассировки запросов

### Ключевые сущности

- **User**: Аккаунт с уникальным email, хэшем пароля (Argon2id), именем, URL аватара, меткой времени создания
- **Character**: D&D-персонаж с именем, классом, расой, характеристиками (JSON), предысторией; принадлежит пользователю
- **Scenario**: Шаблон приключения с названием, тоном, сложностью, ссылкой на создателя; имеет версии
- **ScenarioVersion**: Неизменяемый снимок с номером версии, контентом (акты, NPC, локации как JSON), пользовательским промптом создания
- **Room**: Игровое лобби со ссылкой на хоста, версию сценария, статусом (waiting/active/completed), максимальным числом игроков
- **RoomPlayer**: Запись присоединения, связывающая комнату, пользователя и выбранного персонажа со статусом готовности
- **GameSession**: Активная игра со ссылкой на комнату, состоянием мира (JSON), меткой времени старта
- **SessionMessage**: Одиночное сообщение со ссылкой на сессию, автором (игрок или DM), ролью, контентом, меткой времени

## Критерии успеха *(обязательно)*

### Измеримые результаты

- **SC-001**: Ответы AI-мастера генерируются и транслируются в течение 10 секунд для 95% запросов
- **SC-002**: API обрабатывает 100 одновременных WebSocket-соединений без деградации
- **SC-003**: Генерация сценария завершается в течение 30 секунд для 90% запросов
- **SC-004**: Извлечение состояния добавляет менее 2 секунд к циклу ответа
- **SC-005**: JWT-аутентификация добавляет менее 50мс задержки к запросам
- **SC-006**: API достигает 99.5% uptime в рабочие часы
- **SC-007**: Средняя стоимость AI на игровую сессию остаётся ниже $1.50 (2-3 часа, 4 игрока)
- **SC-008**: Кэширование промптов снижает затраты на AI минимум на 40%
- **SC-009**: API проходит аудит безопасности без критических уязвимостей
- **SC-010**: Все эндпоинты возвращают соответствующие коды ошибок и сообщения

## Допущения

- AI-провайдер (Anthropic Claude) используется для всех AI-операций
- Модель Sonnet для ответов мастера и генерации сценариев; Haiku для извлечения состояния и валидации
- PostgreSQL для постоянного хранения; Redis для кэша сессий и pub/sub
- S3-совместимое хранилище (R2) для аудиофайлов
- JWT-токены с истечением 7 дней, refresh-токены с истечением 30 дней
- WebSocket-соединения используют per-room каналы для трансляции
- Rate limiting: 60 запросов/минуту на пользователя для REST, без лимита для WebSocket-сообщений
- Модерация контента полагается преимущественно на функции безопасности AI-провайдера
- Русский язык — основной; AI-промпты и ответы на русском
