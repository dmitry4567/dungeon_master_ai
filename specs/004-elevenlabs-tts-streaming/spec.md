# Feature Specification: ElevenLabs TTS Streaming

**Feature Branch**: `004-elevenlabs-tts-streaming`
**Created**: 2026-02-27
**Status**: Draft
**Input**: Интеграция ElevenLabs streaming API для потокового воспроизведения текста сообщений DM в реальном времени с возможностью остановки воспроизведения

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Прослушивание сообщения DM (Priority: P1)

Пользователь хочет прослушать текстовое сообщение от Dungeon Master, чтобы погрузиться в атмосферу игры и не читать длинные описания самостоятельно. Он нажимает кнопку "Прослушать" на сообщении, и аудио начинает воспроизводиться немедленно (первые фрагменты), пока остальной текст продолжает обрабатываться на сервере.

**Why this priority**: Это основная ценность функции — мгновенное потоковое воспроизведение без ожидания полной генерации аудио. Пользователь получает аудио практически сразу.

**Independent Test**: Можно протестировать нажатием кнопки "Прослушать" на любом сообщении DM и проверкой, что аудио начинает воспроизводиться в течение 2-3 секунд.

**Acceptance Scenarios**:

1. **Given** открыта игровая сессия с сообщениями от DM, **When** пользователь нажимает кнопку "Прослушать" на сообщении DM, **Then** аудио начинает воспроизводиться в течение 3 секунд
2. **Given** аудио воспроизводится, **When** поступают новые фрагменты аудио, **Then** воспроизведение продолжается без пауз и заиканий
3. **Given** сообщение DM длинное (более 500 символов), **When** пользователь запускает прослушивание, **Then** аудио начинает воспроизводиться до полной генерации всего текста

---

### User Story 2 - Остановка воспроизведения (Priority: P1)

Пользователь хочет иметь возможность остановить воспроизведение в любой момент, если ему надоело слушать или он хочет перейти к другому сообщению.

**Why this priority**: Критически важно для UX — пользователь должен иметь полный контроль над воспроизведением.

**Independent Test**: Запустить воспроизведение, нажать кнопку остановки, проверить что аудио останавливается немедленно.

**Acceptance Scenarios**:

1. **Given** аудио сообщения воспроизводится, **When** пользователь нажимает кнопку остановки (та же кнопка), **Then** воспроизведение немедленно прекращается
2. **Given** аудио воспроизводится, **When** пользователь останавливает воспроизведение, **Then** соединение с сервером корректно закрывается и ресурсы освобождаются
3. **Given** воспроизведение остановлено, **When** пользователь нажимает "Прослушать" снова, **Then** аудио начинает воспроизводиться с начала сообщения

---

### User Story 3 - Визуальная индикация состояния (Priority: P2)

Пользователь хочет видеть текущее состояние воспроизведения (играет/не играет) чтобы понимать, активно ли сейчас аудио.

**Why this priority**: Важно для UX, но функция работоспособна и без индикации.

**Independent Test**: Запустить воспроизведение и проверить, что кнопка изменила внешний вид (иконка/анимация).

**Acceptance Scenarios**:

1. **Given** сообщение DM отображается, **When** воспроизведение не активно, **Then** отображается кнопка с иконкой "Прослушать" (speaker/volume)
2. **Given** воспроизведение запущено, **When** аудио воспроизводится, **Then** кнопка меняется на иконку "Стоп" или отображает анимацию воспроизведения
3. **Given** аудио загружается (буферизация), **When** ожидание первых данных, **Then** отображается индикатор загрузки

---

### User Story 4 - Воспроизведение только одного сообщения (Priority: P2)

Если пользователь запускает прослушивание другого сообщения, текущее воспроизведение должно автоматически остановиться.

**Why this priority**: Предотвращает наложение аудио и путаницу.

**Independent Test**: Запустить воспроизведение одного сообщения, затем нажать "Прослушать" на другом — первое должно остановиться.

**Acceptance Scenarios**:

1. **Given** воспроизводится аудио сообщения A, **When** пользователь нажимает "Прослушать" на сообщении B, **Then** воспроизведение A останавливается и начинается воспроизведение B
2. **Given** одновременно может воспроизводиться только одно сообщение, **When** новое воспроизведение запущено, **Then** предыдущий аудио-поток корректно закрыт

---

### Edge Cases

- Что происходит при потере интернет-соединения во время воспроизведения? (Воспроизведение прерывается, отображается сообщение об ошибке)
- Что если ElevenLabs API недоступен? (Кнопка показывает ошибку, пользователь может попробовать позже)
- Что если превышен rate limit API? (Показать "Превышен лимит прослушиваний, попробуйте позже (через минуту)")
- Что если исчерпана квота/баланс ElevenLabs? (Показать "Озвучка временно недоступна", скрыть кнопки TTS)
- Что если воспроизведение прервано входящим звонком или Siri? (Приостановить, автоматически продолжить после завершения прерывания)
- Как система обрабатывает очень короткие сообщения (менее 10 символов)? (Воспроизводятся нормально)
- Что если пользователь выходит из игровой сессии во время воспроизведения? (Воспроизведение автоматически останавливается)
- Как обрабатываются сообщения с эмодзи и специальными символами? (Текст очищается перед отправкой в TTS)
- Что если пользователь быстро нажимает кнопку несколько раз? (Дебаунс предотвращает множественные запросы)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять кнопку "Прослушать" на каждом сообщении от DM
- **FR-002**: Система ДОЛЖНА отправлять текст сообщения на сервер при нажатии кнопки
- **FR-003**: Сервер ДОЛЖЕН проксировать запрос к ElevenLabs API с потоковой передачей
- **FR-004**: Клиент ДОЛЖЕН воспроизводить аудио-фрагменты по мере их поступления
- **FR-005**: Пользователь ДОЛЖЕН иметь возможность остановить воспроизведение в любой момент
- **FR-006**: Система ДОЛЖНА отображать текущее состояние воспроизведения (играет/не играет/загружается)
- **FR-007**: При запуске нового воспроизведения система ДОЛЖНА остановить предыдущее
- **FR-008**: Система ДОЛЖНА корректно освобождать ресурсы при остановке воспроизведения
- **FR-009**: Сервер ДОЛЖЕН требовать аутентификацию для доступа к TTS-эндпоинту
- **FR-010**: Система ДОЛЖНА обрабатывать ошибки сети и API с информативными сообщениями
- **FR-011**: Сервер ДОЛЖЕН очищать текст от Markdown-разметки перед отправкой в ElevenLabs API
- **FR-012**: Система НЕ ДОЛЖНА показывать кнопку TTS на сообщениях игроков (только DM)
- **FR-013**: Система ДОЛЖНА автоматически разбивать длинные сообщения на части при превышении лимита API и воспроизводить их последовательно без пауз
- **FR-014**: При rate limit система ДОЛЖНА показывать "Превышен лимит прослушиваний, попробуйте позже (через минуту)"
- **FR-015**: При исчерпании квоты/баланса API система ДОЛЖНА показывать "Озвучка временно недоступна" и скрывать кнопки TTS
- **FR-016**: Система ДОЛЖНА продолжать воспроизведение аудио когда приложение находится в фоновом режиме
- **FR-017**: При прерывании аудио-сессии (входящий звонок, Siri) система ДОЛЖНА приостановить воспроизведение и автоматически продолжить после завершения прерывания

### Key Entities

- **TTSState**: Состояние воспроизведения TTS (содержит TTSStatus: idle, loading, playing, paused, error) и связанный ID сообщения
- **TTSRequest**: Запрос на генерацию речи (messageId, text, voice_id)
- **TTSAudioChunk**: Фрагмент аудио-данных в потоке (binary audio data)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Первый фрагмент аудио начинает воспроизводиться в течение 3 секунд после нажатия кнопки
- **SC-002**: Воспроизведение останавливается в течение 500мс после нажатия кнопки остановки
- **SC-003**: Нет слышимых пауз или заиканий при воспроизведении потокового аудио
- **SC-004**: 95% пользователей успешно прослушивают сообщения без ошибок
- **SC-005**: Функция работает на устройствах с различным качеством интернет-соединения (3G+)
- **SC-006**: Пользователи понимают состояние воспроизведения по визуальной индикации

## Assumptions

- ElevenLabs API-ключ будет храниться только на сервере (клиент не имеет прямого доступа к API)
- Голос для TTS — женский голос, выбран заранее и не требует выбора пользователем (можно добавить позже)
- Формат аудио — MP3 для максимальной совместимости
- Язык озвучки — русский (по умолчанию), соответствует языку игры
- Потоковая передача будет осуществляться через HTTP streaming (Server-Sent Events или chunked response)

## Clarifications

### Session 2026-02-27

- Q: Как обрабатывать длинные сообщения, превышающие лимит API? → A: Автоматически разбивать на части и воспроизводить последовательно
- Q: Как реагировать на лимиты API? → A: Rate limit — "Превышен лимит прослушиваний, попробуйте позже (через минуту)"; исчерпание квоты/баланса — "Озвучка временно недоступна"
- Q: Должно ли воспроизведение продолжаться в фоновом режиме? → A: Да, продолжать воспроизведение в фоне
- Q: Какой характер голоса для Dungeon Master? → A: Голос девушки
- Q: Как реагировать на прерывание аудио (звонок, Siri)? → A: Приостановить, автоматически продолжить после завершения прерывания

## Out of Scope

- Выбор голоса пользователем
- Кэширование сгенерированного аудио
- Офлайн-воспроизведение
- Настройка скорости воспроизведения
- Воспроизведение сообщений игроков
- TTS для стримящихся сообщений (только для завершённых сообщений DM)
