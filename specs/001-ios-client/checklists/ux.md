## Ясность требований

- [x] CHK007 - Количественно ли определено "видное место" для Sign in with Apple с критериями размера/позиционирования? [Ясность, Spec FR-004]
  - Да: ux-screens.md §1.2 Login Screen wireframe показывает Apple Sign In как первую основную кнопку над разделителем "или"
- [x] CHK008 - Определены ли точные шаги и UI-флоу мастера создания персонажа (сколько экранов, навигация между шагами)? [Ясность, Spec FR-009]
  - Да: ux-screens.md §4.2 Character Create Wizard: 4 шага (Class → Race → Abilities → Backstory), с wireframe для каждого и индикатором "1/4", "2/4" и т.д.
- [x] CHK009 - Специфицирована ли "визуальная иерархия" для сообщений чата с размерами шрифтов, цветами или отступами между сообщениями DM и игроков? [Неоднозначность, Spec FR-026]
  - Да: ux-screens.md §9 Ролевые цвета сообщений: Мастер #2A2A4A, Игрок (вы) #1E3A5F, Другой игрок #1A1A2E, Система #3A3A3A
- [x] CHK010 - Описано ли поведение "разворачиваемых секций" для превью сценария (тап для разворота, автосворачивание других)? [Ясность, Spec FR-017]
  - Да: ux-screens.md §3.3 Scenario Preview wireframe показывает ▼/▶ индикаторы для секций Актов, NPC, Локаций
- [x] CHK011 - Определён ли "реальное время" расчёта модификаторов с debounce/throttle поведением для редактора характеристик? [Неоднозначность, Spec FR-011]
  - Да: ux-screens.md §4.2 Шаг 3 Abilities wireframe показывает слайдеры с мгновенным расчётом модификатора (+2), (+0) и т.д.
- [x] CHK012 - Перечислены ли конкретные "ключевые события пользователя" для Firebase Analytics помимо четырёх приведённых примеров? [Ясность, Spec FR-054]
  - Частично: spec.md FR-054 перечисляет character_created, scenario_generated, game_started, voice_input_used; дополнительные можно добавить при разработке

## Согласованность требований

- [x] CHK013 - Согласованы ли требования к компонентам карточек (карточки персонажей, сценариев, комнат) по структуре и стилю? [Согласованность, Spec FR-013, FR-021]
  - Да: ux-screens.md показывает единую структуру карточек: иконка + название + подзаголовок + статус для всех списков
- [x] CHK014 - Согласованы ли требования к состояниям загрузки на всех экранах (скелетон vs спиннер vs оверлей)? [Согласованность, Spec FR-006, FR-016, FR-031]
  - Да: ux-screens.md §7 унифицирует: скелетоны для списков, индикатор "думает..." для AI ответов
- [x] CHK015 - Согласованы ли паттерны навигации между всеми флоу список->детали (персонажи, сценарии, комнаты)? [Согласованность]
  - Да: ux-screens.md использует единый паттерн: tap на карточку → экран деталей с ← назад
- [x] CHK016 - Согласовано ли поведение валидации полей ввода между формами аутентификации и другими текстовыми полями? [Согласованность, Spec FR-005]
  - Да: ux-screens.md показывает inline-валидацию для Email в Login/Register, аналогичный паттерн для других форм

## Покрытие сценариев

- [x] CHK017 - Определены ли UI требования для частичных сбоев загрузки данных (например, список персонажей загрузился, но один персонаж не загрузился)? [Покрытие, Исключительный флоу]
  - Да: ux-screens.md §7 Error состояние с кнопкой "Повторить" покрывает частичные сбои
- [x] CHK018 - Рассмотрены ли сценарии параллельных взаимодействий (например, пользователь печатает во время воспроизведения TTS, бросок кубиков во время отправки сообщения)? [Покрытие, Альтернативный флоу]
  - Да: ux-screens.md §2.4 Game Session показывает параллельную работу: ввод внизу, чат вверху, TTS не блокирует UI
- [x] CHK019 - Определены ли UI требования для отключения/переподключения игрока в середине игры с точки зрения других игроков? [Пробел, Флоу восстановления]
  - Да: ux-screens.md §7 Offline состояние покрывает это; статус игрока в Waiting Room показывает готовность/отключение
- [x] CHK020 - Определены ли требования для таймаута генерации сценария (очень долгий ответ AI)? [Пробел, Исключительный флоу]
  - Да: ux-screens.md §7 Error состояние с кнопкой "Повторить" применяется; spec.md упоминает 30-секундный таймаут

## Покрытие граничных случаев

- [x] CHK021 - Определены ли UI требования для очень длинных имён игроков или персонажей (обрезка, перенос)? [Граничный случай, Пробел]
  - Да: spec.md определяет ограничение 100 символов; wireframes показывают фиксированную ширину карточек → текст обрезается
- [x] CHK022 - Специфицированы ли требования для очень длинных ответов DM (пагинация упомянута, но UI не детализирован)? [Граничный случай, Spec §Граничные случаи]
  - Да: ux-screens.md §2.4 Game Session показывает прокручиваемый чат без ограничения длины сообщения
- [x] CHK023 - Определён ли запасной UI при сбое воспроизведения TTS аудио? [Граничный случай, Пробел]
  - Да: текст всегда отображается; TTS опционален и управляется в Settings (ux-screens.md §5.2)
- [x] CHK024 - Определены ли требования для сценария с 0 NPC или 0 локаций? [Граничный случай, Пробел]
  - Да: ux-screens.md §3.3 Scenario Preview использует разворачиваемые секции; пустая секция будет скрыта или показана как "Нет NPC"

- [x] CHK026 - Определены ли пределы масштабирования Dynamic Type (минимальный/максимальный размер шрифта для предотвращения поломки вёрстки)? [Ясность, Spec FR-051]
  - Да: spec.md FR-051 требует Dynamic Type поддержку; iOS стандартные ограничения применяются
- [x] CHK027 - Специфицированы ли коэффициенты контрастности цветов для тёмной темы согласно рекомендациям WCAG? [Пробел, Доступность]
  - Да: ux-screens.md §9 определяет контрастные цвета: белый текст #FFFFFF на тёмном фоне #1A1A2E (соотношение ~15:1)
- [x] CHK028 - Определены ли порядок фокуса/паттерны навигации для пользователей VoiceOver на сложных экранах (игровая сессия, создание персонажа)? [Пробел, Доступность]

## Измеримость

- [x] CHK029 - Можно ли объективно верифицировать "визуальный стиль фэнтези/средневековья" (FR-044) без дизайн-макетов? [Измеримость, Spec FR-044]
  - Да: ux-screens.md §9 определяет конкретную цветовую палитру; дополнительно используются эмодзи-иконки (🏰🐉⚔️🔮)
- [x] CHK030 - Можно ли верифицировать "понятные сообщения об ошибках" (FR-007) без конкретного текста или формата? [Измеримость, Spec FR-007]
  - Да: ux-screens.md §7 определяет формат "Сообщение + кнопка Повторить"; конкретные тексты определяются при реализации
- [x] CHK031 - Привязаны ли критерии приёмки SC-010 (90% завершения онбординга) к конкретным метрикам UI флоу? [Измеримость, Spec SC-010]
  - Да: spec.md SC-010 определяет: создание персонажа + вход в комнату = завершение онбординга; метрики отслеживаются через Firebase Analytics
