## Пользовательские сценарии и тестирование *(обязательно)*

### Пользовательская история 1 — Интерфейс игровой сессии (Приоритет: P1)

Игрок видит экран игровой сессии с интерфейсом в стиле чата, отображающим нарратив приключения. Он может вводить или произносить действия, видеть запросы на броски кубиков от мастера, бросать кубики с анимацией и слышать/читать ответы мастера в реальном времени вместе с другими игроками.

**Почему этот приоритет**: Это основной пользовательский опыт — интерфейс взаимодействия с AI-мастером. Без этого приложение не имеет ценности.

**Независимый тест**: Можно протестировать, открыв игровую сессию, отправив 3 действия (1 текстом, 1 голосом), бросив кубики по запросу и проверив корректное отображение всех ответов.

**Сценарии приёмки**:

1. **Дано** игрок в активной игровой сессии, **Когда** он смотрит на экран, **Тогда** он видит прокручиваемый чат с действиями игроков и ответами мастера в хронологическом порядке
2. **Дано** игрок вводит действие, **Когда** он нажимает отправить, **Тогда** действие появляется в чате и индикатор загрузки показывается в ожидании ответа мастера
3. **Дано** ответ мастера с запросом броска кубиков, **Когда** отображается, **Тогда** появляется интерфейс броска кубиков с правильным типом кубика, модификатором и сложностью
4. **Дано** активный бросок кубиков, **Когда** игрок нажимает бросить, **Тогда** проигрывается анимация кубика и результат (с модификаторами) показывается и отправляется в сессию
5. **Дано** голосовой ввод включён, **Когда** игрок удерживает кнопку микрофона, **Тогда** речь транскрибируется и показывается в поле ввода для проверки перед отправкой
6. **Дано** TTS включён, **Когда** приходит ответ мастера, **Тогда** аудио воспроизводится автоматически, а текст прокручивается синхронно

### Пользовательская история 2 — Интерфейс конструктора сценариев (Приоритет: P2)

Пользователь открывает экран конструктора сценариев, где может описать идею приключения в текстовом поле, просмотреть сгенерированную структуру сценария (акты, NPC, локации), запросить изменения через дополнительные промпты и применить сценарий к комнате.

**Почему этот приоритет**: Это позволяет создавать контент, что является ключевым отличием. Интерфейс должен делать AI-сгенерированные сценарии осязаемыми и редактируемыми.

**Независимый тест**: Можно протестировать, создав сценарий, просмотрев превью, сделав одну правку и применив к комнате.

**Сценарии приёмки**:

1. **Дано** пользователь на экране конструктора сценариев, **Когда** он вводит описание и нажимает сгенерировать, **Тогда** показывается состояние загрузки, затем структурированное превью с разворачиваемыми секциями
2. **Дано** превью сценария, **Когда** пользователь нажимает на акт/NPC/локацию, **Тогда** полные детали разворачиваются на месте
3. **Дано** превью сценария, **Когда** пользователь вводит промпт уточнения и нажимает обновить, **Тогда** затронутые секции подсвечиваются и обновляются
4. **Дано** сценарий с версиями, **Когда** пользователь открывает историю версий, **Тогда** показывается список предыдущих версий с метками времени и промптами
5. **Дано** финализированный сценарий, **Когда** пользователь нажимает "Применить к комнате", **Тогда** его переводят на создание комнаты с предвыбранным сценарием

---

### Пользовательская история 3 — Интерфейс создания персонажа (Приоритет: P3)

Пользователь создаёт персонажа через пошаговый мастер: выбор класса, выбор расы, назначение характеристик (с подсказками), написание предыстории и сохранение. Интерфейс показывает особенности класса/расы и валидирует по правилам D&D 5e.

**Почему этот приоритет**: Персонажи нужны для игры. Интерфейс должен направлять новых игроков через создание персонажа D&D.

**Независимый тест**: Можно протестировать, создав персонажа от начала до конца и использовав его в игровой комнате.

**Сценарии приёмки**:

1. **Дано** пользователь начинает создание персонажа, **Когда** он выбирает класс, **Тогда** отображаются особенности класса и рекомендуемые характеристики
2. **Дано** класс выбран, **Когда** пользователь выбирает расу, **Тогда** показываются расовые бонусы и особенности, рекомендации по характеристикам обновляются
3. **Дано** экран назначения характеристик, **Когда** пользователь назначает значения, **Тогда** модификаторы рассчитываются и отображаются в реальном времени
4. **Дано** все обязательные поля заполнены, **Когда** пользователь сохраняет, **Тогда** персонаж появляется в списке персонажей с карточкой-сводкой
5. **Дано** список персонажей, **Когда** пользователь нажимает на персонажа, **Тогда** отображается полный лист персонажа со всеми характеристиками и предысторией

---

### Пользовательская история 4 — Интерфейс игрового лобби (Приоритет: P4)

Пользователь просматривает доступные игровые комнаты, создаёт новую комнату со сценарием, приглашает игроков, управляет запросами на вступление и начинает игру, когда все готовы.

**Почему этот приоритет**: Координация мультиплеера важна, но может следовать за основным геймплеем.

**Независимый тест**: Можно протестировать, создав комнату, дождавшись присоединения другого игрока и начав игру.

**Сценарии приёмки**:

1. **Дано** пользователь на экране лобби, **Когда** просматривает, **Тогда** список открытых комнат показывает название сценария, имя хоста и количество игроков
2. **Дано** пользователь создаёт комнату, **Когда** он выбирает сценарий и нажимает создать, **Тогда** комната создаётся и пользователь показывается как хост в комнате ожидания
3. **Дано** хост в комнате ожидания, **Когда** приходит запрос на вступление, **Тогда** появляется уведомление с именем игрока и кнопками принять/отклонить
4. **Дано** игроки в комнате ожидания, **Когда** каждый игрок отмечает готовность, **Тогда** его аватар показывает индикатор готовности
5. **Дано** все игроки готовы, **Когда** хост нажимает старт, **Тогда** все игроки переходят на экран игровой сессии одновременно

---

### Пользовательская история 5 — Интерфейс аутентификации (Приоритет: P5)

Пользователь видит экран входа/регистрации при первом запуске с кнопкой Sign in with Apple и опцией email/пароль. После аутентификации он получает доступ к основному приложению. Сессия сохраняется между перезапусками приложения.

**Почему этот приоритет**: Требуется для доступа, но применяются стандартные паттерны интерфейса.

**Независимый тест**: Можно протестировать, зарегистрировавшись, закрыв приложение, открыв заново и проверив автоматический вход.

**Сценарии приёмки**:

1. **Дано** первый запуск приложения, **Когда** пользователь видит экран авторизации, **Тогда** кнопка Sign in with Apple отображается на видном месте с опцией email ниже
2. **Дано** нажата Sign in with Apple, **Когда** авторизация Apple завершается, **Тогда** пользователь переводится на главный экран
3. **Дано** регистрация по email, **Когда** пользователь вводит корректный email/пароль, **Тогда** аккаунт создаётся и пользователь переходит на главный экран
4. **Дано** авторизованный пользователь, **Когда** он открывает приложение после закрытия, **Тогда** он автоматически входит и видит главный экран

---

### Пользовательская история 6 — Интерфейс профиля и истории (Приоритет: P6)

Пользователь просматривает свой профиль с аватаром, именем и настройками. Он может видеть своих персонажей, сценарии и историю завершённых игр.

**Почему этот приоритет**: Профиль вторичен по отношению к основному геймплею, но нужен для управления пользователем.

**Независимый тест**: Можно протестировать, просмотрев профиль, изменив имя и просмотрев историю игр.

**Сценарии приёмки**:

1. **Дано** пользователь на экране профиля, **Когда** просматривает, **Тогда** отображаются аватар, имя, email и настройки
2. **Дано** экран профиля, **Когда** пользователь нажимает вкладку персонажей, **Тогда** показывается список его персонажей
3. **Дано** экран профиля, **Когда** пользователь нажимает вкладку сценариев, **Тогда** показывается список созданных им сценариев
4. **Дано** экран профиля, **Когда** пользователь нажимает вкладку истории, **Тогда** показывается список завершённых игр с датой и названием сценария

### Граничные случаи

- Что происходит при потере сетевого соединения во время игровой сессии?
  - Приложение показывает баннер "Переподключение..." и ставит в очередь исходящие сообщения; при переподключении синхронизирует пропущенные сообщения
- Как приложение обрабатывает переходы фон/передний план во время игры?
  - WebSocket отключается в фоне; переподключается на переднем плане с синхронизацией сообщений
- Что происходит, когда распознавание голоса не возвращает текст?
  - Поле ввода остаётся пустым; пользователь может повторить или ввести вручную
- Как приложение обрабатывает очень длинные ответы мастера?
  - По умолчанию: текст прокручивается в чате без пагинации
  - TTS воспроизводится полностью; пользователь может прокручивать независимо от воспроизведения
  - Автопрокрутка к концу при завершении TTS (если пользователь не прокручивал вручную)
- Что происходит при прерывании анимации броска кубиков?
  - Результат всё равно применяется; пользователь видит финальный результат без анимации

### Ключевые сущности (клиентская сторона)

- **UserProfile**: Локальное представление авторизованного пользователя с аватаром, именем, кэшированными настройками
- **Character**: Локально кэшированные данные персонажа с классом, расой, характеристиками для оффлайн-просмотра
- **Scenario**: Локально кэшированная структура сценария для превью и редактирования
- **Room**: Состояние комнаты в реальном времени, синхронизируемое через WebSocket
- **GameSession**: Активное состояние сессии с историей сообщений, снимком состояния мира
- **Message**: Одиночное сообщение чата с автором, контентом, меткой времени, опциональным результатом броска

## Критерии успеха *(обязательно)*

### Измеримые результаты

- **SC-001**: Пользователи могут перемещаться между всеми основными экранами за 2 нажатия
- **SC-002**: Мастер создания персонажа завершается менее чем за 5 минут для новых пользователей
- **SC-003**: Превью сценария загружается в течение 2 секунд после завершения генерации
- **SC-004**: Анимация броска кубиков завершается в течение 1.5 секунд
- **SC-005**: Транскрипция голоса появляется в поле ввода в течение 1 секунды после окончания речи
- **SC-006**: Приложение поддерживает 60fps при прокрутке и анимациях игровой сессии
- **SC-007**: Холодный старт приложения до главного экрана менее 3 секунд
- **SC-008**: Размер приложения остаётся менее 100МБ для первоначальной загрузки
- **SC-009**: Приложение проходит ревью Apple App Store с первой попытки
- **SC-010**: 90% пользователей завершают онбординг (первый персонаж + первое игровое действие) без отсева
  - Измеряется через Firebase Analytics воронку: регистрация → создание персонажа → первое действие в игре
